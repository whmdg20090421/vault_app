# 加密文件处理模块技术需求（最终完整版V4） Spec

## Why
原有的加密流处理存在诸多问题（如导入文件夹任务无法正确展开、大小统计错误、UI 刷新与后台执行脱节等）。为了彻底解决这些问题并提升应用稳定性和用户体验，按照用户提供的“最终完整版V4”技术需求，重新设计并实现加密任务的数据模型、多线程引擎、状态机以及动态进度刷新与 UI 呈现。

## What Changes

### 一、文件导入与路径获取
用户点击「导入加密文件夹」或「导入加密文件」按钮，唤起系统原生文件选择界面；用户选择单个文件或文件夹并确认后，系统将所选绝对路径提交至应用核心处理模块，创建独立加密任务。

### 二、加密文件结构树定义与构建
#### 2.1 整体结构规范
- 每个加密任务对应一个独立的根节点（用户选择的单个文件或文件夹）
- 所有加密任务的根节点统一存储在全局任务列表中，采用JSON格式持久化至应用内部私有数据目录
- 每个根节点递归向下展开生成完整层级文件树

#### 2.2 节点字段定义
- 文件夹节点：仅用于维护层级结构
  - `taskId`：全局唯一任务ID（仅根节点有）
  - `name`：文件夹/文件名称
  - `type`：固定值 `folder`/`file`
  - `isPaused`：布尔值，标记该节点及其子节点是否被手动暂停
  - `children`：子节点数组（仅文件夹节点有）
- 文件节点：存储文件核心信息与加密状态
  - `name`：完整文件名（包含文件后缀）
  - `type`：固定值 `file`
  - `size`：文件大小（自动换算单位，保证数值在1~1000之间，保留1位小数，单位依次为B、KB、MB、GB、TB）
  - `rawSize`：文件原始字节大小（用于进度计算，保证精度）
  - `status`：加密状态枚举值
    - `pending_waiting`：未加密·等待中（含未开始、加密中断）
    - `pending_paused`：未加密·已暂停（用户手动暂停）
    - `encrypting`：正在加密
    - `completed`：已加密完成
    - `error`：错误·等待修复（文件意外缺失或损坏）
  - `isPaused`：布尔值，继承父节点的暂停状态

#### 2.3 构建流程
1. 接收用户提交的路径，生成全局唯一 `taskId`，创建对应根节点并加入全局任务列表
2. 递归遍历根节点下所有子文件与子文件夹
3. 对每个文件节点，同步读取原始字节大小并自动换算为规范格式
4. 所有新建节点默认 `isPaused=false`，文件节点默认初始化为 `pending_waiting` 状态
5. 构建完成后，将更新后的全局任务列表写入应用内部私有数据目录
6. 新任务创建后，自动触发一次全局进度刷新，并启动动态刷新机制

### 三、多线程加密执行
1. 读取应用设置中预设的加密线程数（支持系统默认值与用户手动配置值），启动对应数量的加密线程并行处理所有任务中** `status=pending_waiting` 且 `isPaused=false` **的文件节点。
2. 加密中断处理规则：
   - 若加密过程中因应用正常退出、用户强制终止或系统异常导致单个文件加密中断，该文件状态将被重置为 `pending_waiting`。
   - 应用重启后，自动读取本地存储的全局任务列表，对所有 `status=pending_waiting` 且 `isPaused=false` 的文件执行完整重新加密，不支持单个文件断点续传。
3. 文件异常处理规则：
   - 加密前/加密中检测到文件意外缺失、损坏或无法读取时，立即终止该文件的加密进程
   - 将该文件节点的 `status` 更新为 `error`
   - 抛出文件异常错误，在UI上标记该文件为异常状态，提示用户手动修复
   - 异常文件不影响其他正常文件的加密进程

### 四、加密状态更新与进度恢复
1. 单个文件开始加密时，立即将结构树中对应节点的 `status` 更新为 `encrypting`。
2. 单个文件加密完成后，立即将对应节点的 `status` 更新为 `completed`。
3. 应用重启时，自动读取本地存储的全局任务列表，基于各节点的 `status` 和 `isPaused` 字段恢复所有加密任务的进度，继续未暂停的未完成任务。
4. 重启时自动扫描所有文件节点，对原文件系统中已不存在的文件，自动将其 `status` 更新为 `error`。

### 五、主页导航栏加密信息总览
#### 5.1 UI 位置与形态
- 位于应用主页顶部导航栏，展示为一个小型四色进度条图标
- 点击时显示详细数值提示

#### 5.2 数据计算逻辑
- 已加密总大小：遍历所有加密任务中所有 `completed` 状态文件的 `rawSize` 之和
- 正在加密总大小：遍历所有加密任务中所有 `encrypting` 状态文件的 `rawSize` 之和
- 待加密总大小：遍历所有加密任务中所有 `pending_waiting` 状态文件的 `rawSize` 之和
- 已暂停/错误总大小：遍历所有加密任务中所有 `pending_paused` 和 `error` 状态文件的 `rawSize` 之和
- 进度条显示：
  - 绿色：已加密占比
  - 黄色：正在加密占比
  - 红色：未加密·等待中占比
  - 灰色：未加密·已暂停+错误·等待修复占比
  - 四者合计100%
- 数值提示：显示 `已加密大小 / 总大小` （自动换算单位）

#### 5.3 进度条强制可视化规则
所有进度条（包括全局总览进度条和条目级进度条）均需遵循：
- 若某一颜色段的理论占比小于1%，强制保持1像素的最小显示长度
- 最小长度不影响其他颜色段的正常比例计算
- 确保用户能够直观感知到所有状态的存在，即使该状态下的文件总大小极小

### 六、层级化加密进度详情查看
#### 6.1 入口与交互
- 主界面提供「查看同步进度」UI按钮，用户点击后，从下往上滑入弹出半屏模态框。
- 模态框内展示所有正在进行的加密任务根节点，各根节点为同级关系（即使原文件系统中不在同一目录）。
- 点击任意任务根节点/文件夹条目可展开/折叠其子目录，支持多级嵌套展开，与实际文件系统层级保持一致。

#### 6.2 条目显示规范（已修正行结构）
每个文件和文件夹条目统一采用三行布局，高度紧凑，避免占用过多垂直空间：
- 第一行（最左侧+名称）：最左侧放置圆形暂停/继续按钮，右侧显示条目名称（文件夹/文件完整名称）；异常状态条目在名称右侧添加红色感叹号图标
- 第二行：单条极细字节进度条
  - 绿色部分：该条目及其所有子目录下已加密文件的总字节占比
  - 黄色部分：该条目及其所有子目录下正在加密文件的总字节占比
  - 红色部分：该条目及其所有子目录下未加密·等待中文件的总字节占比
  - 灰色部分：该条目及其所有子目录下未加密·已暂停+错误·等待修复文件的总字节占比
  - 四者合计100%
  - 严格遵循5.3节的进度条强制可视化规则
- 第三行（统计信息行）：
  - 左侧：显示当前条目整体加密百分比（保留1位小数，如 `67.3%` ）；异常状态条目显示「存在异常」
  - 右侧：显示加密数量统计
    - 文件条目：固定显示 `1/1`
    - 文件夹/任务根节点条目：显示 `已加密文件数/总文件数` （仅统计该条目及其所有子目录下的文件总数，不包含文件夹本身）

#### 6.3 暂停/继续功能（条目级）
1. 按钮状态与交互：
   - 正常运行状态：按钮显示「暂停图标」
   - 暂停状态：按钮显示「继续图标」
2. 暂停范围：点击任意条目的暂停按钮，立即终止该条目及其所有子节点下所有正在加密的线程，并将该条目及其所有子节点的 `isPaused` 字段设为 `true`，所有 `encrypting` 状态的文件重置为 `pending_paused`。
3. 继续范围：点击任意条目的继续按钮，将该条目及其所有子节点的 `isPaused` 字段设为 `false`，所有 `pending_paused` 状态的文件重新加入加密线程池待处理队列，状态更新为 `pending_waiting`。
4. 优先级规则：父节点暂停会覆盖所有子节点的状态；子节点暂停不影响父节点和同级节点的运行状态。

#### 6.4 长按删除功能（正在进行的任务）
1. 触发方式：长按任意条目（文件/文件夹/任务根节点），底部弹出操作菜单，显示「移除加密」选项；异常状态条目额外显示「标记为已修复」选项。
2. 删除范围：
   - 仅删除被长按的该节点及其所有子节点，同级节点和父节点的加密状态完全不受影响
   - 示例：根节点A包含子目录a-1、a-2、a-3，长按a-1并删除后，a-1及其所有子文件被移除加密任务，a-2、a-3继续正常加密
3. 删除后处理：
   - 立即终止该节点下所有正在运行的加密线程，释放系统资源
   - 从全局任务列表中永久移除该节点及其所有子节点
   - 实时更新全局进度统计和所有相关条目的进度显示
4. 确认机制：点击「移除加密」后弹出二次确认对话框，防止误操作。
5. 异常修复功能：点击「标记为已修复」后，将该节点及其所有子节点的 `error` 状态重置为 `pending_waiting`，重新加入加密队列。

### 七、动态进度刷新机制（核心优化）
#### 7.1 基本定义
- 前台显示状态：任何需要展示进度相关数据的界面处于打开状态，包括但不限于：
  - 应用主页（显示顶部导航栏进度条）
  - 同步进度详情模态框
  - 任何需要读取加密结构树、文件大小、加密百分比的界面
- 后台静默状态：所有上述界面均已关闭，应用处于后台或无进度展示需求的状态

#### 7.2 有活跃加密任务时的刷新规则
1. 后台静默状态：
   - 刷新频率调整为每5~10秒随机一次（避免多应用同时唤醒导致的系统资源竞争）
   - 仅更新内存中的全局进度数据，不触发任何UI重绘
   - 每次刷新后将最新数据持久化至本地存储
2. 前台显示状态：
   - 当用户打开任何需要展示进度的界面时，立即执行一次全量进度刷新
   - 随后切换为每0.5~1秒随机一次的高频刷新模式
   - 每次刷新同时更新内存数据和所有可见UI元素
3. 状态切换规则：
   - 当最后一个需要展示进度的界面关闭时，立即从高频刷新切换回后台低频刷新
   - 当任何需要展示进度的界面打开时，立即从后台低频刷新切换回前台高频刷新

#### 7.3 无活跃加密任务时的刷新规则
1. 当所有加密任务均完成（所有文件节点 `status=completed`）时，立即执行一次最终全量刷新
2. 将最终进度数据永久缓存至本地存储
3. 停止所有自动刷新任务，不再进行任何周期性进度计算
4. 当用户打开任何需要展示进度的界面时，直接读取缓存的最终数据进行展示
5. 只有当创建新的加密任务时，才重新启动动态刷新机制

#### 7.4 独立监听线程职责
- 仅负责维护全局进度数据的计算和更新，不直接操作UI
- 根据当前应用状态自动调整刷新频率
- 确保在任何情况下，进度数据的持久化操作都能正常完成

### 八、任务完成与自动归档
#### 8.1 任务完成判定
当某个加密任务根节点下所有文件节点的 `status` 均为 `completed` 时，判定该任务完成。

#### 8.2 自动归档流程
1. 任务完成后，立即执行目录存在性检查：遍历该任务根节点下所有文件和文件夹，验证其在原文件系统中是否仍然存在。
2. 若所有文件和文件夹均存在，则将该任务从「正在进行的加密任务列表」中移除，移至加密历史记录中永久保存。
3. 若存在文件/文件夹缺失，则保留该任务在「正在进行的加密任务列表」中，将缺失文件的 `status` 更新为 `error`，并在进度详情中标记异常状态，提示用户手动处理。
4. 当所有任务均完成并归档后，触发7.3节的无活跃任务刷新规则。

#### 8.3 加密历史记录管理
1. 历史记录列表展示所有已完成的加密任务，条目布局与正在进行的任务一致（仅隐藏暂停按钮和进度条，显示完成时间）。
2. 长按删除功能：长按任意历史记录条目，底部弹出「删除历史」选项，二次确认后从历史记录列表中永久移除该任务。
3. 删除历史记录仅清除任务进度数据，不删除已加密的文件和本地哈希索引库，不影响后续文件解密和完整性校验。

### 九、加密完成后处理
单个文件加密完成后，除更新结构树状态外，还需执行以下操作：
计算该加密后文件的哈希值，将哈希值与文件唯一标识（`taskId + 相对路径`）关联写入本地哈希索引库，用于后续文件完整性校验、重复文件识别及解密验证等场景。

## Impact
- Affected specs: 加密任务流、多线程调度、状态机、进度更新、持久化归档与索引写入。
- Affected code:
  - `lib/encryption/models/encryption_node.dart` (数据模型，结构树定义)
  - `lib/encryption/services/encryption_task_manager.dart` (全局进度管理、监听线程、调度与自动归档)
  - `lib/encryption/ui/` (主页导航栏进度条、半屏进度模态框等相关UI组件)
