# 加密文件处理模块技术需求 V4 Spec

## Why
原有的加密流处理存在诸多问题（如导入文件夹任务无法正确展开、大小统计错误、UI 刷新与后台执行脱节等）。为了彻底解决这些问题并提升应用稳定性和用户体验，按照“最终完整版V4”技术需求，重新设计并实现加密任务的数据模型、多线程引擎、状态机以及动态进度刷新与 UI 呈现。

## What Changes
- **数据模型重构**：定义基于树状结构（`folder` / `file` 节点）的全局加密任务模型，使用 `taskId` 作为根节点标识，持久化存储为 JSON。
- **状态机与异常流**：为文件节点引入 `pending_waiting`, `pending_paused`, `encrypting`, `completed`, `error` 5 种状态，处理异常中断、重置及断点恢复逻辑。
- **多线程加密引擎**：根据设置中的线程数，启动多线程并行处理 `status=pending_waiting` 且 `isPaused=false` 的节点。
- **动态进度刷新机制**：引入独立监听线程，根据前台（0.5~1秒）/后台（5~10秒）/无活跃任务的不同状态，动态切换全量进度刷新与持久化频率。
- **总览与详情 UI**：
  - 导航栏四色进度条（已加密/正在加密/等待/暂停或错误），强制 1 像素可视化规则。
  - 下滑弹出的半屏模态框，展示层级化树状进度，支持三行紧凑布局（包含极细字节进度条和百分比/数量统计）。
  - 支持条目级暂停/继续（级联控制）与长按删除/异常修复。
- **归档与后处理**：任务完成后进行目录存在性校验，移至历史记录，并计算哈希值写入本地哈希索引库。

## Impact
- Affected specs: 核心加密任务流、多线程调度、状态机、进度通知与 UI 渲染。
- Affected code:
  - `lib/encryption/models/` (新建数据模型)
  - `lib/encryption/services/encryption_task_manager.dart` (核心引擎重构)
  - `lib/encryption/ui/` (新建模态框、进度条 UI 组件)
  - `lib/encryption/vault_explorer_page.dart` (入口与总览集成)

## ADDED Requirements
### Requirement: 树状任务结构与动态刷新
系统必须将加密任务构建为层级树，并根据前后台状态智能刷新进度。

#### Scenario: 导入嵌套文件夹并查看进度
- **WHEN** 用户导入一个包含子文件夹的目录，并打开查看进度模态框
- **THEN** 系统将其构建为包含 `folder` 和 `file` 节点的树状 JSON，自动启动多线程加密。模态框以前台高频（0.5~1s）刷新进度，各条目显示极细的四色进度条（严格遵循 1% 以下强制 1 像素规则），并支持点击展开/折叠层级。

### Requirement: 条目级暂停与级联控制
系统必须支持对任意节点进行暂停或继续，并正确级联影响子节点。

#### Scenario: 暂停父文件夹
- **WHEN** 用户在模态框中点击某个父文件夹的“暂停”按钮
- **THEN** 该文件夹及其所有子节点的 `isPaused` 设为 `true`，所有 `encrypting` 的文件转为 `pending_paused` 并终止对应线程，且不影响同级其他文件夹的正常加密。

### Requirement: 任务完成归档与哈希写入
系统必须在任务完成后进行完整性校验并记录哈希。

#### Scenario: 加密任务顺利完成
- **WHEN** 某根节点下所有文件节点状态均为 `completed`
- **THEN** 系统校验文件在原文件系统中存在后，将其移出活跃队列并存入历史记录，同时计算加密后文件的哈希值，关联 `taskId + 相对路径` 写入本地哈希索引库。
