【开发规范 - 严格执行，不得违反】

1. 禁止假数据与占位符
   任何数据展示类组件（图表、统计、列表、进度条等）必须接入真实数据源。
   严禁使用硬编码假数据、Math.random()、固定数值作为展示内容。
   若真实数据暂时为空，允许显示"暂无数据"空态，但不得伪造数据填充。

2. 禁止未实现的功能入口
   严禁创建任何点击后无响应、弹出"功能未开放"、"即将上线"、
   "开发中"等提示的按钮、菜单项或入口。
   若某功能当前无法完整实现，宁可不创建该入口，也不允许创建空壳入口。

3. 功能必须真实可用
   每一个交付的功能模块，必须完成从数据层到 UI 层的完整实现链路。
   UI 组件必须与真实业务逻辑绑定，不允许只完成视觉层而跳过逻辑层。
   例如：加密进度条必须读取真实加密状态；文件统计必须扫描真实文件；
   任务列表必须来自真实任务队列，而不是预置的虚拟列表项。

4. 实现优先于美观
   在功能实现与 UI 美观发生冲突时，优先保证功能的真实可用性。
   允许 UI 简陋，不允许功能是空壳。

5. 报错不等于未完成
   功能实现后如果存在 bug 或报错，应通过后续调试修复，
   但不允许以"避免报错"为理由将功能替换为假数据或空实现。
   必须写出真实的实现逻辑，哪怕它可能在首次运行时存在问题。（若报错的输出在我前面指定的报错目录里面，包括尽量的使用中文。）

6. 更新日志强制追加
   维护更新日志 (CHANGELOG) 时必须采用“只能追加 (Append-only)”的方式。
   严禁修改或删除历史版本的更新记录，新版本的更新内容必须添加到文件顶部或末尾（保持原有时间线顺序）。

7. 进度与待办跟踪
   开发是知道你的每次会阅读的区域的，因此，任何**目前只创造了 UI 但未实现具体功能的**模块，或**有计划开发的**需求，必须统一记录在 `.trae/DEVELOPMENT_PLAN.md`（开发日志）中。每次会话开始时应结合此文件了解待办进度。
